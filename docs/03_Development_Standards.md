# Phase 2 - Development Standards & Technology Stack

> **Implementation status (Sprint 20):** This is a planning-era standards document. The current implementation uses React component state rather than Zustand and does not include Docker files. For the deployable configuration and verified commands, use the root README and [Testing and Deployment Guide](09_Testing_Deployment.md).

## Purpose

This document defines the technology stack, development standards, coding practices, project structure, and engineering principles for the Car Dealership Inventory Management System.

The goal is to establish a consistent development environment before implementation begins.

# 2. Backend Architecture

The backend follows a Hybrid Clean Architecture.

Version 1 Application Flow:

API

↓

Repository

↓

SQLAlchemy ORM

↓

PostgreSQL

Lightweight business logic and validation remain in API routes; repositories are limited to database access. A Service layer is deferred for Version 1.

This improves:

- Maintainability
- Testability
- Scalability
- Code readability

# 5. Engineering Principles

The project follows the following engineering principles:

- Write clean and readable code.
- Prefer simplicity over unnecessary complexity.
- Every engineering decision should have a business justification.
- Follow consistent naming conventions.
- Keep business logic outside API routes.
- Maintain modular and scalable architecture.
- Write code that is easy to test.
- Design for future scalability while keeping Version 1 focused.

# 6. Architecture Decisions

## ADR-001

Framework:

FastAPI

Reason:

High performance, asynchronous support, automatic API documentation, recruiter familiarity.

---

## ADR-002

Database:

PostgreSQL

Reason:

Reliable production-grade relational database with excellent SQLAlchemy support.

---

## ADR-003

ORM:

SQLAlchemy 2.0

Reason:

Industry-standard ORM with strong integration into FastAPI.

---

## ADR-004

Architecture:

Hybrid Clean Architecture

Reason:

Provides clear separation of responsibilities while avoiding unnecessary complexity for a medium-sized project.

---

## ADR-005

State Management:

Zustand

Reason:

Simple, lightweight, scalable, and significantly cleaner than Context API for authentication and shared application state.

---

## ADR-006

HTTP Client:

Axios

Reason:

Cleaner API calls, request interceptors, centralized error handling, and easier JWT token management.

# Phase 2 Summary

The technology stack has been finalized using modern, industry-standard tools that prioritize maintainability, scalability, security, and recruiter familiarity.

The project follows a Hybrid Clean Architecture with clear separation between presentation, business logic, and data access layers.

This foundation ensures that implementation remains organized, testable, and extensible while keeping Version 1 focused on delivering a production-quality inventory management system.
